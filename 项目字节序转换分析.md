### 1.2 自定义字节序转换函数

项目中实现了自己的字节序转换函数，**始终使用大端序（网络字节序）**：

#### GlobalMessage.h中的实现：
```cpp
class CIMsg {
public:
    // 写入数据：主机序 → 大端序
    static void WriteData(BYTE *pBuffer, DWORD dwData) {
        if (pBuffer) {
            BYTE hhByte = (BYTE) ((dwData & 0xff000000) >> 24);  // 最高字节
            BYTE hlByte = (BYTE) ((dwData & 0x00ff0000) >> 16);
            BYTE lhByte = (BYTE) ((dwData & 0x0000ff00) >> 8);
            BYTE llByte = (BYTE) ((dwData & 0x000000ff));        // 最低字节
            
            // 按大端序存储：高字节在前
            pBuffer[0] = hhByte;  // 最高字节存储在最低地址
            pBuffer[1] = hlByte;
            pBuffer[2] = lhByte;
            pBuffer[3] = llByte;  // 最低字节存储在最高地址
        }
    }
    
    // 读取数据：大端序 → 主机序
    static DWORD ReadData(BYTE *pBuffer) {
        DWORD dwValue = 0;
        if (pBuffer) {
            BYTE hhByte = pBuffer[0];  // 最高字节
            BYTE hlByte = pBuffer[1];
            BYTE lhByte = pBuffer[2];
            BYTE llByte = pBuffer[3];  // 最低字节
            
            // 按大端序解析
            dwValue = ((DWORD)(hhByte) << 24) +    // 最高字节左移24位
                     ((DWORD)hlByte << 16) +
                     ((DWORD)lhByte << 8) +
                     ((DWORD)llByte);              // 最低字节不移位
        }
        return dwValue;
    }
};
```

## 2. 字节序转换的必要性

### 2.1 网络通信标准

**网络字节序 = 大端序**

所有网络协议（TCP/IP、UDP等）都使用大端序作为标准：
- IP地址
- 端口号
- 协议头中的长度字段
- 自定义协议的数据字段

### 2.2 跨平台兼容性

不同CPU架构使用不同的字节序：

| 架构 | 字节序 | 说明 |
|------|--------|------|
| x86/x64 | 小端序 | Intel、AMD处理器 |
| ARM (默认) | 小端序 | 大多数ARM处理器 |
| PowerPC | 大端序 | IBM处理器 |
| MIPS | 可配置 | 可以是大端或小端 |
| 网络 | 大端序 | 网络字节序标准 |

### 2.3 数据一致性

在你的项目中，不同终端可能运行在不同架构上：
- 发送端：x86 Linux（小端序）
- 接收端：ARM嵌入式设备（小端序）
- 路由节点：PowerPC设备（大端序）

如果不进行字节序转换，数据会被错误解析。

```

├──────────────────────────────────────────┤

htonl(2500000) = 0x002625A0 (网络序)
```

## 5. 为什么需要字节序转换？

### 5.1 实际例子

假设要传输数值 `0x12345678`：

**小端序系统（x86）：**
```
内存布局：[0x78] [0x56] [0x34] [0x12]
         低地址 ←----------→ 高地址
```

**大端序系统（PowerPC）：**
```
内存布局：[0x12] [0x34] [0x56] [0x78]
         低地址 ←----------→ 高地址
```

**网络传输（大端序）：**
```
网络字节流：0x12 0x34 0x56 0x78
```

### 5.2 不转换的后果

如果小端序系统直接发送 `0x12345678`：

```
发送端（小端）：直接发送内存内容
网络字节流：0x78 0x56 0x34 0x12

接收端（大端）：按大端序解析
解析结果：0x78563412  ← 错误！
```

### 5.3 正确的转换

```cpp
// 发送端（小端系统）
uint32_t value = 0x12345678;
uint32_t network_value = htonl(value);  // 转换为网络序
send(socket, &network_value, 4, 0);

// 接收端（大端系统）
uint32_t network_value;
recv(socket, &network_value, 4, 0);
uint32_t host_value = ntohl(network_value);  // 转换为主机序
// host_value = 0x12345678  ← 正确！
```

## 6. 项目中的字节序策略

### 6.1 统一使用大端序

项目采用了**统一的大端序策略**：

1. **网络传输**：使用标准的网络字节序（大端序）
2. **协议封装**：自定义协议头使用大端序
3. **数据存储**：持久化数据使用大端序
4. **内存处理**：运行时转换为主机序

### 6.2 转换点

```
应用数据（主机序）
    ↓ WriteData() / htonl()
协议封装（大端序）
    ↓
网络传输（大端序）
    ↓
协议解析（大端序）
    ↓ ReadData() / ntohl()
应用数据（主机序）
```

### 6.3 一致性保证

所有模块都使用相同的转换函数：
- `CIMsg::WriteData/ReadData`（GlobalMessage.h）
- `Term2TermQos::WriteData/ReadData`（DataCodec.cpp）
- `htonl/ntohl`（标准库）

## 7. 总结

### 7.1 项目确实使用字节序转换

你的项目**大量使用了字节序转换**，主要体现在：

1. **视频反馈包**：使用 `htonl/ntohl` 转换统计数据
2. **协议头部**：使用自定义函数转换终端ID、长度等
3. **QoS数据**：使用自定义函数转换丢包率、速率等
4. **网络地址**：使用 `htons` 转换端口号

### 7.2 转换的必要性

- **跨平台兼容**：支持不同架构的设备通信
- **网络标准**：遵循TCP/IP协议栈的字节序规范
- **数据一致性**：确保数据在不同系统间正确解析

### 7.3 实现方式

- **标准函数**：`htonl/ntohl/htons/ntohs`
- **自定义函数**：`WriteData/ReadData`
- **运行时检测**：动态检测系统字节序

### 7.4 设计优点

1. **统一性**：所有模块使用相同的转换策略
2. **可靠性**：避免了字节序不匹配导致的数据错误
3. **兼容性**：支持异构网络环境
4. **标准化**：遵循网络编程最佳实践

你的项目在字节序处理方面做得很规范，这是网络通信程序的重要组成部分！
